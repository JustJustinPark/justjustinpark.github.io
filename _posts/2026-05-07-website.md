## Coding challenges

With a design vision and layout in mind, I went on the hunt for suitable [Jekyll](https://jekyllrb.com/) themes.
I eventually decided on [Chirpy](https://github.com/cotes2020/jekyll-theme-chirpy) for its minimalist aesthetic and vertical navigation bar, although it was clear that I would need to make a few tweaks in order to get it to where I wanted.
Easier said than done, apparently.

(For the record, I think Chirpy is a great theme with solid developer support, and it's the closest I found to my target design. I definitely recommend it if you're looking to create a blog-focused site with a similar overall layout!)

In this inaugural blog post, I wanted to highlight two of the main issues I ran into while trying to modify the template.
Ironically enough, the first issue was that I didn't like the prominence of the blog!
More specifically, I wanted a home page that existed separately from the blog, whereas Chirpy is built so that the landing page *is* the blog paginator.
Fortunately, I found [this helpful guide](https://jinchaoli.com/blog/exchange-homepage-and-about) by a fellow Chirpy user with the same goal, so I followed the provided steps while incorporating a few of my own ideas as well.
The most significant changes I made were as follows:
- Repurposed the existing "About" page as a homepage and set its permalink to `/`
- Configured the permalink for all blog-related tabs (the paginator, "Categories", "Tags", and "Archive") to start with `/blog/`
- Added additional LiquidJS logic in `_includes/sidebar.html` to group the blog-related navigation tabs in the sidebar

The other main issue had to do with CSS, because of course it did. In the appropriate stylesheet within the Chirpy template, I originally defined the following CSS variables and style declarations:

```css
:root {
  --sidebar-text-color-active: #255820;
  --sidebar-hover-bg-color: #cceebb55;
}

#sidebar .nav-link {
  --sidebar-text-color-active: #255820;
  --sidebar-hover-bg-color: #cceebb55;
}
```

However, after building the site and poring through the computed styles in my browser's DevTools panel, I found the following active declarations:

```css
#sidebar ul li.nav-item.active .nav-link {
  color: var(--sidebar-active-color);
  background-color: var(--sidebar-hover-bg);
}
```

Okay, so those aren't the variables I defined...
These seem to be coming from the Chirpy template stylesheet, rather than my own custom styles.
But my styles are declared after the template's, so shouldn't mine take priority?

I'm sure that anyone who regularly writes CSS already spotted the issue or saw this coming a mile away, and unfortunately I am not one of those people.
If you're like me and don't work with CSS often, feel free to pause for a moment and ponder what's going on.

<hr>

All right, welcome back!

In short, the problem is one of **specificity**.
For those unfamiliar, in CSS there is a concept called the "specificity hierarchy" which determines which style declaration wins out over another.
It's basically a priority order, and different types of selectors take priority over others: 
1. Inline style declarations automatically have the highest priority and override all other styles. (Since Jekyll is Markdown-based, there are rarely)
2. ID selectors such as `#header` are next in the hierarchy and provide the next-highest priority.
3. Class selectors such as `.caption` and attribute selectors such as `:hover` come next. (This level also includes pseudo-classes. I mention these separately because I honestly don't know what a CSS pseudo-class is.)
4. Towards the bottom of the hierarchy are elements such as `h1` (and pseudo-elements, like `::after`).
5. And finally, the universal selector `*` and the `:where()` pseudo-class lose to everything else.

Importantly, in a combination of selectors, each of #2&ndash;4 contribute to an overall specificity "score" or "weight":

| Selector type                  | Weight |
| :----------------------------- | :----: |
| ID                             | 1-0-0  |
| Class, pseudo-class, attribute | 0-1-0  |
| Elements, pseudo-elements      | 0-0-1  |
{: .with-border}

As a concrete example, let's revisit the template's style declaration from earlier, specifically its selectors:

```css
#sidebar ul li.nav-item.active .nav-link {
  ...
}
```

We see that this consists of one ID selector (`#sidebar`), three class selectors (`.nav-item`, `.active`, and `.nav-link`), and two element selectors (`ul` and `li`), for a total specificity weight of 1-3-2.
(It's perhaps worth noting that I originally discovered the concept of specificity while trying to debug this issue, when I hovered over these selectors in the DevTools panel only to be shown some hovertext stating its specificity weight. Neat!)

So, the solution for my issue? Simply write out a more specific style declaration, like so:

```css
#sidebar ul li.nav-item.active a.nav-link {
  background-color: var(--sidebar-hover-bg-color);
  color: var(--sidebar-text-color-active);
}
```

This has a weight of 1-3-3, thus taking priority over the existing template style.
(A weight of 1-3-2 would also suffice, since later declarations override earlier ones when their specificity weights match.)
